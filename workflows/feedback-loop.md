# 피드백 루프 가이드 (Feedback Loop Guide)

## 개요

검증 실패 시 어떻게 이전 단계로 돌아가 수정할지 정의한 가이드.

---

## 피드백 루프 원칙

### 원칙 1: 최대 3회 루프
- 같은 문제로 3회 이상 반복 시 사람(개발자) 개입
- 무한 루프 방지

### 원칙 2: 근본 원인 파악
- 단순 수정이 아닌 원인 분석
- 반복 방지

### 원칙 3: 모든 루프 기록
- `logs/feedback-loops/` 폴더에 저장
- 패턴 분석 가능

### 원칙 4: 상위 단계로 먼저 복귀
- 회계 로직 오류는 회계사 에이전트로
- 코드 품질은 백엔드 에이전트로

---

## 루프 라우팅 매트릭스

### 백엔드 검증 실패 시

| 문제 유형 | 복귀 단계 | 이유 |
|---|---|---|
| K-IFRS 조항 인용 오류 | 1️⃣ 회계사 | 회계 지식 재확인 |
| 계산 로직 오류 (회계) | 1️⃣ 회계사 | 요구사항 재정의 |
| 계산 로직 오류 (구현) | 2️⃣ 백엔드 | 코드 수정 |
| 엣지 케이스 누락 | 1️⃣ 회계사 | 케이스 재정의 |
| 코드 품질 | 2️⃣ 백엔드 | 리팩토링 |
| 테스트 부족 | 2️⃣ 백엔드 | 테스트 추가 |
| 보안 이슈 | 2️⃣ 백엔드 | 수정 |
| 감사 추적 누락 | 2️⃣ 백엔드 | 추가 구현 |

### 프론트 검증 실패 시

| 문제 유형 | 복귀 단계 | 이유 |
|---|---|---|
| UI/UX 문제 | 4️⃣ 프론트 | 디자인 수정 |
| API 연동 오류 | 4️⃣ 프론트 | 연동 수정 |
| 타입 에러 | 4️⃣ 프론트 | 타입 수정 |
| 백엔드 API 자체 문제 | 2️⃣ 백엔드 | API 수정 |
| 시나리오 불일치 | 1️⃣ 회계사 | 요구사항 재확인 |

### 최종 검증 실패 시

| 문제 유형 | 복귀 단계 |
|---|---|
| 회계 로직 미스 | 1️⃣ 회계사 |
| 백엔드 구현 미스 | 2️⃣ 백엔드 |
| UI 미스 | 4️⃣ 프론트 |
| 문서 부족 | 6️⃣ 문서화 |
| 데모 시나리오 불일치 | 전체 재검토 |

## 루프 유형별 상세

### 타입 1: 회계 로직 오류

#### 증상
- K-IFRS 조항 잘못 인용
- 계산 공식 부정확
- 엣지 케이스 오해

#### 경로
검증 실패
↓
회계사 에이전트 (재정의)
↓
백엔드 에이전트 (재구현)
↓
백엔드 검증 (재검증)

#### 명령 예시
"최종 검증에서 공정가치 헤지 분개가 K-IFRS 6.5.8과
불일치한다고 나왔어.
회계사 에이전트로서 다시 분석해줘:

- RAG에서 1109호 6.5.8 재검색
- 요구사항 수정
- requirements/fair-value-hedge.md 업데이트"

---

### 타입 2: 구현 오류

#### 증상
- 코드 품질 문제
- 테스트 실패
- BigDecimal 미사용
- Null 처리 부족

#### 경로
검증 실패
↓
백엔드 에이전트 (재구현)
↓
백엔드 검증 (재검증)

#### 명령 예시

"백엔드 검증에서 다음 문제 지적됨:

- FairValueService에서 double 사용 (BigDecimal로 수정)
- 테스트 커버리지 60% (80%로 올리기)
- Javadoc 누락

백엔드 에이전트로서 수정해줘.
수정 범위: backend/src/.../service/FairValueService.java"

---

### 타입 3: UI 오류

#### 증상
- 디자인 부적절
- API 연동 실패
- 반응형 미지원

#### 경로
프론트 검증 실패
↓
프론트 에이전트 (재구현)
↓
프론트 검증 (재검증)

---

### 타입 4: 문서 불일치

#### 증상
- API 문서와 실제 동작 차이
- 사용자 매뉴얼 부정확

#### 경로
최종 검증 실패
↓
문서화 에이전트 (재작성)
↓
최종 검증 (재검증)

---

## 루프 기록 방법

### 파일 구조
logs/feedback-loops/
├── [기능명]_[날짜].log
│   예: fair-value-fx-forward_2026-04-20.log

### 기록 템플릿
```markdown
# 피드백 루프 기록: [기능명]

## 루프 #1
- **일시**: 2026-04-20 10:30
- **단계**: 백엔드 검증 → 회계사
- **이슈**: K-IFRS 6.5.11과 6.5.12 혼동
- **수정 내용**: 
  - 요구사항에서 OCI vs P/L 명확히 구분
  - 6.5.11 (유효 부분)
  - 6.5.12 (비유효 부분)
- **결과**: 백엔드 재구현 → 통과

## 루프 #2
- **일시**: 2026-04-20 14:00
- **단계**: 프론트 검증 → 프론트
- **이슈**: 금액 표시 포맷 오류 (천 단위 콤마 없음)
- **수정 내용**: Intl.NumberFormat 적용
- **결과**: 통과

## 총 루프 수: 2회
## 최종 상태: APPROVED
```

## 루프 예시 시나리오

### 시나리오 1: K-IFRS 조항 오류 (3회 루프)
[1차 시도]
회계사 → 백엔드 → 검증 (FAIL: 6.5.11 인용 오류)
[루프 1: 회계사로 복귀]
회계사 → RAG 재검색 → 수정 → 백엔드 → 검증 (FAIL: 여전히 오류)
[루프 2: 회계사 상세 재검토]
회계사 → 감사인 관점 검토 → 수정 → 백엔드 → 검증 (FAIL)
[루프 3: 사람(개발자) 개입]
"3회 루프 초과. 개발자가 직접 K-IFRS 조항 확인 필요."
→ 개발자가 원문 확인 → 정정 → 회계사 에이전트에 피드백
→ 올바른 조항 사용 → 통과 ✅

### 시나리오 2: 간단한 수정 (1회 루프)
백엔드 → 검증 (FAIL: Javadoc 누락)
→ 백엔드 (Javadoc 추가)
→ 검증 (PASS) ✅

## 루프 횟수 관리

### 카운터
```python
# 유사코드
loop_count = {
  "accounting_loop": 0,
  "backend_loop": 0,
  "frontend_loop": 0,
  "documentation_loop": 0
}

MAX_LOOP = 3
```

### 초과 시 처리
if loop_count >= 3:

1. 사람(개발자)에게 알림
2. logs/에 상세 기록
3. 파이프라인 일시 중단
4. 근본 원인 분석 후 재개

## 루프 방지 전략

### 1. 명확한 요구사항
- 회계사 에이전트가 처음부터 정확히
- RAG 검색 충분히
- 엣지 케이스 미리 고려

### 2. 강한 검증 기준
- PASS 기준 명확
- 애매하면 FAIL
- 회색지대 최소화

### 3. 피드백 구체성
- "코드 품질 문제" X
- "FairValueService line 45 double → BigDecimal" O

### 4. 컨텍스트 유지
- 이전 시도 내용 기억
- 같은 실수 반복 방지

## 특수 케이스

### Case 1: 검증 에이전트 간 의견 불일치

상황: 백엔드 검증은 PASS, 최종 검증은 FAIL

처리:
1. 최종 검증의 지적이 우선
2. 백엔드 검증 기준 재검토
3. 검증 에이전트 정의 개선

### Case 2: 순환 피드백

상황:
- 회계사: "백엔드 잘못"
- 백엔드: "요구사항 잘못"

처리:
1. 사람(개발자) 중재
2. RAG 원문 재확인
3. 요구사항 vs 구현 차이 분석
4. 명확한 결론 후 재개

### Case 3: 새로운 K-IFRS 해석 발견

상황: 개발 중 더 나은 조항 발견

처리:
1. 회계사 에이전트로 전환
2. RAG 재검색
3. 요구사항 업데이트
4. 처음부터 재시작

## 사람(개발자) 개입 기준

### 즉시 개입 상황
- 루프 3회 초과
- 명백한 K-IFRS 오해
- 데이터 손실 위험
- 보안 취약점 발견

### 리뷰 권장 상황
- 루프 2회 경험
- 애매한 회계 판단
- 새로운 엣지 케이스
