## 1. 헥사고날 아키텍처 리팩토링

### 배경

기존 레이어드 아키텍처(`controller → service → repository`)는 컨트롤러가 Service 구현체를 직접 주입받아, 도메인 로직과 인프라 의존성이 뒤섞이는 문제가 있었다. 백엔드 검증 에이전트 1차 검증에서 "컨트롤러가 UseCase 인터페이스를 사용하지 않고 Service 구현체를 직접 주입"을 이유로 NEEDS_REWORK 판정을 받아 전면 전환했다.

### 패키지 구조

```
com.hedge.prototype/
├── domain/               # 순수 도메인 — Spring Bean 없음, @Service 금지
│   ├── hedge/            # HedgeRelationship, HedgedItemType, InstrumentType, ...
│   ├── valuation/        # FxForwardPricing, IrsPricing, CrsPricing, ...
│   ├── effectiveness/    # DollarOffsetCalculator, LowerOfTestCalculator, ...
│   └── journal/          # JournalEntry, JournalEntryType, CashFlowHedgeJournalGenerator, ...
│
├── application/          # 애플리케이션 계층 — UseCase + Service + Event
│   ├── hedge/            # HedgeCommandUseCase (인터페이스) + HedgeDesignationService
│   ├── valuation/        # FxForwardValuationUseCase, IrsValuationUseCase, CrsValuationUseCase
│   ├── effectiveness/    # EffectivenessTestUseCase + EffectivenessTestService
│   ├── journal/          # JournalEntryUseCase + JournalEntryService + PdfExporter
│   ├── event/            # Spring 내부 이벤트 (designated / valuated / tested)
│   └── port/             # 아웃바운드 포트 (Repository 인터페이스)
│       ├── hedge/
│       ├── valuation/
│       ├── effectiveness/
│       └── journal/
│
└── adapter/
    └── web/              # 인바운드 어댑터 — Controller + DTO
        ├── hedge/
        ├── valuation/
        ├── effectiveness/
        └── journal/
```

### 핵심 변경 원칙

- **Controller는 UseCase 인터페이스만 주입** (`@RequiredArgsConstructor` + 인터페이스 타입 필드)
- **Service 구현체는 Controller에서 보이지 않음** (인터페이스 뒤로 은닉)
- **domain/ 패키지는 Spring 의존성 0** — `@Slf4j`(Lombok)만 허용, `@Service`, `@Component` 금지
- **Repository 인터페이스는 `{domain}/application/port/` 에 위치** — JPA 구현체(`*RepositoryImpl`)는 별도 infra 어댑터에서 처리

### 변경 전후 비교

| 항목 | 변경 전 (레이어드) | 변경 후 (헥사고날) |
|------|-------------------|-------------------|
| 컨트롤러 의존 대상 | `*ServiceImpl` (구현체) | `*UseCase` (인터페이스) |
| 도메인 Spring 의존성 | 일부 `@Service` 혼재 | 완전 분리 |
| 테스트 용이성 | 구현체 직접 모킹 필요 | UseCase Mock 주입 가능 |
| 아웃바운드 포트 위치 | `repository/` 디렉토리 | `{domain}/application/port/` 패키지 |

## 2. Spring 내부 이벤트 구현

### 이벤트 설계 원칙

- `@TransactionalEventListener(phase = BEFORE_COMMIT)` 패턴 적용
- **Zero-Payload 원칙**: 이벤트 레코드에는 ID만 전달, 핸들러에서 DB 재조회
  - 이유: 이벤트 발행 시점과 핸들러 실행 시점의 데이터 일관성 보장
- 핸들러 실패 시 발행자 트랜잭션도 함께 롤백 (BEFORE_COMMIT 보장)

### 구현된 이벤트 3종

#### 2-1. HedgeDesignatedEvent

```java
// hedge/application/event/HedgeDesignatedEvent.java
public record HedgeDesignatedEvent(
        String hedgeRelationshipId,
        HedgeType hedgeType
) {}
```

- 발행 시점: 헤지 지정(Designation) 완료 후
- 용도: 지정 후 처리 (향후 문서 자동 생성, 알림 연동 예정)

#### 2-2. ValuationCompletedEvent

```java
// valuation/application/event/ValuationCompletedEvent.java
public record ValuationCompletedEvent(
        Long valuationId,
        String hedgeRelationshipId
) {}
```

- 발행 시점: 공정가치 평가 완료 후
- 용도: 평가 완료 알림, 연계 유효성 테스트 트리거 (향후)

#### 2-3. EffectivenessTestCompletedEvent

```java
// effectiveness/application/event/EffectivenessTestCompletedEvent.java
public record EffectivenessTestCompletedEvent(
        Long effectivenessTestId,
        String hedgeRelationshipId,
        EffectivenessTestResult testResult,
        ActionRequired actionRequired,
        HedgeType hedgeType
) {}
```

- 발행 시점: 유효성 테스트 완료 후
- 핵심 역할: **테스트 결과에 따른 자동 분개 생성 트리거**

### EffectivenessTestCompletedEventHandler 처리 흐름

```
유효성 테스트 완료
    ↓ BEFORE_COMMIT 이벤트 발행
EffectivenessTestCompletedEventHandler.handle()
    ├── testResult == FAIL → 분개 생략, 경고 로그
    ├── actionRequired == DISCONTINUE → 분개 생략, 수동 처리 필요 로그
    └── PASS / WARNING
            ↓ effectivenessTestId로 DB 재조회
            ├── hedgeType == FAIR_VALUE
            │       → JournalEntryRequest.forAutoGenerationFvh() 팩토리 호출
            └── hedgeType == CASH_FLOW
                    → JournalEntryRequest.forAutoGenerationCfh() 팩토리 호출
                    (effectiveAmount, ineffectiveAmount 포함)
            ↓
        JournalEntryUseCase.createEntries() 호출
```

K-IFRS 근거:
- K-IFRS 1109호 6.4.4 (유효성 요건)
- K-IFRS 1109호 6.5.8 (공정가치 위험회피 회계처리)
- K-IFRS 1109호 6.5.11 (현금흐름 위험회피 OCI 인식)

### 3-3. 자동 분개 버그 수정

**문제**: 이전 CFH 이벤트 핸들러에서 `effectiveAmount`와 `ineffectiveAmount`가 null로 전달되어, 자동 분개 생성 시 OCI 금액이 0으로 처리되거나 NPE가 발생했다.

**원인**: 단일 팩토리 메서드 `forAutoGeneration()`이 hedgeType에 무관하게 동일한 파라미터를 받아, CFH에 필수인 effectiveAmount/ineffectiveAmount를 누락했다.

**해결**: 팩토리 메서드를 FVH/CFH로 분리

```java
// FVH 전용 팩토리 — effectiveAmount, ineffectiveAmount 불필요
public static JournalEntryRequest forAutoGenerationFvh(
        String hedgeRelationshipId,
        LocalDate entryDate,
        BigDecimal instrumentFvChange,
        BigDecimal hedgedItemFvChange) { ... }

// CFH 전용 팩토리 — effectiveAmount, ineffectiveAmount 명시적 요구
public static JournalEntryRequest forAutoGenerationCfh(
        String hedgeRelationshipId,
        LocalDate entryDate,
        BigDecimal instrumentFvChange,
        BigDecimal hedgedItemFvChange,
        BigDecimal effectiveAmount,
        BigDecimal ineffectiveAmount) { ... }

// 기존 메서드는 @Deprecated 처리 (하위 호환)
@Deprecated
public static JournalEntryRequest forAutoGeneration(...) { ... }
```

## 4. 새 헤지 대상 상품 추가 (HedgedItemType enum)

K-IFRS 1109호 6.3.1(위험회피대상항목 적격성) 및 6.3.7(위험 구성요소 지정)에 근거하여, 증권사 실무에서 빈번한 6종 헤지대상항목 유형을 enum으로 정의했다.

### 추가된 HedgedItemType 목록

| enum | 한글명 | 허용 헤지위험 | 허용 헤지유형 | K-IFRS 근거 |
|------|--------|-------------|-------------|------------|
| `FOREIGN_BOND` | 외화채권 | FOREIGN_CURRENCY, INTEREST_RATE | FAIR_VALUE, CASH_FLOW | 1109호 6.3.7 |
| `KRW_FIXED_BOND` | 원화 고정금리채권 | INTEREST_RATE | FAIR_VALUE | 1109호 6.5.8 |
| `KRW_FLOATING_BOND` | 원화 변동금리채권 | INTEREST_RATE | CASH_FLOW | 1109호 6.5.11 |
| `FOREIGN_BORROWING` | 외화차입금 | FOREIGN_CURRENCY, INTEREST_RATE | FAIR_VALUE, CASH_FLOW | 1109호 6.3.1 |
| `FORECAST_TRANSACTION` | 외화 예상거래 | FOREIGN_CURRENCY | CASH_FLOW | 1109호 6.3.1 |
| `FX_DEPOSIT` | 외화예금 | FOREIGN_CURRENCY | FAIR_VALUE, CASH_FLOW | 1109호 6.3.1 |

### 설계 특이사항

- 각 enum 값이 `allowedRisks[]`와 `allowedHedgeTypes[]`를 내장
- `isRiskAllowed(HedgedRisk)`, `isHedgeTypeAllowed(HedgeType)` 메서드로 지정 시 즉시 검증 가능
- 배열은 방어적 복사(`clone()`)로 불변성 보장
- `FORECAST_TRANSACTION`은 CASH_FLOW 전용 — K-IFRS 1109호 6.3.1에 따라 예상거래는 FVH 불가

## 5. IRS (이자율스왑) 구현

### 5-1. 도메인 엔티티

**IrsContract** (`valuation/domain/irs/IrsContract.java`)
- 계약 ID, 명목금액(KRW, BigDecimal), 고정금리, 변동금리지수(SOFR, LIBOR 등), 결제주기
- Pay/Receive 방향: `payFixedReceiveFloating` 필드 (boolean)
  - `true`: Pay Fixed / Receive Floating (변동금리 차입 CFH에 사용)
  - `false`: Pay Floating / Receive Fixed (고정금리채권 FVH에 사용)

**IrsValuation** (`valuation/domain/irs/IrsValuation.java`)
- Append-Only 설계 — 한 번 저장된 평가 이력은 수정/삭제 없음 (K-IFRS 감사 추적)
- 평가 결과: `fixedLegPv`, `floatingLegPv`, `fairValue`

### 5-2. IrsPricing 계산 로직

```java
// 순수 도메인 클래스 — Spring Bean 아님, DB 없음
public class IrsPricing {

    // 할인계수: df = 1 / (1 + r × T/365)  [ACT/365 기준]
    public static BigDecimal calculateDiscountFactor(BigDecimal discountRate, int days) { ... }

    // 고정 다리 PV = fixedRate × notional × Σdf_i
    // 결제주기별 분할: QUARTERLY=4회/년, SEMI_ANNUAL=2회/년, ANNUAL=1회/년
    public static BigDecimal calculateFixedLegPv(
            BigDecimal notional, BigDecimal fixedRate,
            int remainingTermDays, String settlementFrequency,
            BigDecimal discountRate) { ... }

    // 변동 다리 PV = currentFloatingRate × notional × df_1기간
    // 단기 IRS 1기간 근사치 방법 (PoC 단순화)
    public static BigDecimal calculateFloatingLegPv(...) { ... }

    // IRS FV (Receive Fixed): 고정PV - 변동PV
    // IRS FV (Pay Fixed):     변동PV - 고정PV
    public static BigDecimal calculateFairValue(
            BigDecimal fixedLegPv, BigDecimal floatingLegPv,
            boolean payFixedReceiveFloating) { ... }
}
```

**PoC 단순화 가정**

- 플랫(Flat) 이자율 커브 — 모든 기간에 동일 할인율 적용
- 변동 다리는 1기간 근사치 (full forward curve 미적용)
- 결제주기별 균등 일수 분할 (실제 날짜 기반 비즈니스 데이 미적용)

K-IFRS 근거: K-IFRS 1113호 72~90항 (Level 2, 관측가능한 투입변수), K-IFRS 1109호 6.5.8

```java
public class CrsPricing {

    public static BigDecimal calculateForeignLegPv(
            BigDecimal foreignNotional, BigDecimal foreignCouponRate,
            BigDecimal spotRate, int remainingTermDays,
            String settlementFrequency, BigDecimal foreignDiscountRate) { ... }

    // 원화 다리 PV

            BigDecimal krwNotional, BigDecimal krwCouponRate,
            int remainingTermDays, String settlementFrequency,
            BigDecimal krwDiscountRate) { ... }

    // CRS FV = 외화다리PV(원화환산) - 원화다리PV

            BigDecimal foreignLegPvKrw, BigDecimal krwLegPv) { ... }
}
```

| 환율 변수 | 불필요 | 필수 (spotRate) |
| 할인율 | 단일 discountRate | 원화/외화 각각 별도 적용 |

K-IFRS 근거: K-IFRS 1113호 72~90항 (Level 2), K-IFRS 1109호 6.5.11, B6.4.9

## 7. HedgeRelationship 헤지수단 다형성

### 변경 내용

기존 `HedgeRelationship`은 `fxForwardContractId` 단일 필드만 있었다. IRS, CRS 추가로 헤지수단이 다양해짐에 따라 다형성 지원이 필요했다.

**추가된 필드**

```java
// hedge/domain/model/HedgeRelationship.java
public class HedgeRelationship {
    // 신규 추가
    private InstrumentType instrumentType;   // FX_FORWARD / IRS / CRS
    private String instrumentId;             // 각 파생상품 계약 ID

    // 기존 필드 — 하위 호환 유지 (FX_FORWARD 시 그대로 사용)
    private String fxForwardContractId;
}
```

```java
public enum InstrumentType {
```

## 9. 백엔드 검증 결과

### 1차 검증 — NEEDS_REWORK

- **판정 이유**: 컨트롤러가 UseCase 인터페이스를 사용하지 않고 Service 구현체를 직접 주입
- **조치**: 헥사고날 아키텍처 전환 (섹션 1 참조)

### 2차 검증 — PASS

```
compileJava  BUILD SUCCESSFUL
test         BUILD SUCCESSFUL

테스트 결과:
- IrsPricingTest  : 17개 PASS
- CrsPricingTest  : 10개 PASS
```

**IrsPricingTest 17개 커버 범위**
- 할인계수 계산 (정상, days=0 경계값)
- 고정 다리 PV (분기/반기/연간 결제주기별)
- 변동 다리 PV (결제주기별)
- IRS 공정가치 (Pay Fixed / Receive Fixed 양방향)
- 지원하지 않는 결제주기 예외 처리

**CrsPricingTest 10개 커버 범위**
- 할인계수 계산
- 외화 다리 PV 원화 환산 (이자 + 원금 재교환)
- 원화 다리 PV (이자 + 원금 재교환)
- CRS 공정가치 (양수/음수)
- 지원하지 않는 결제주기 예외 처리

## 10. 오늘 발생한 주요 이슈 및 해결

| 백엔드 1차 검증 실패 | Controller → ServiceImpl 직접 주입 | 헥사고날 아키텍처 전환 |
