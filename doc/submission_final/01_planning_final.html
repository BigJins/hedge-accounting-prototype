01 Goal and Background

목표와 배경

위험회피회계는 계약 정보, 시장 데이터, 회계 기준, 유효성 테스트, 분개 근거가 함께 맞아야 적용할 수 있습니다. 수작업 중심의 엑셀 처리에서는 계산 방식과 판단 근거가 사람마다 달라질 수 있고, 감사 대응 시 어떤 기준으로 분개가 생성됐는지 추적하기 어렵습니다.

01
해결하려는 문제

공정가치 평가, 헤지 지정, 유효성 테스트, 자동 분개가 서로 끊겨 있어 반복 입력과 검증 누락이 발생하는 문제를 줄입니다.

02
필요한 이유

유가증권 운용사는 환위험과 금리위험을 지속적으로 관리해야 하며, 보고기간 말마다 회계처리와 근거를 빠르게 확인해야 합니다.

03
프로토타입 목표

K-IFRS 1109호와 1113호 기준을 반영해 입력부터 분개 확인까지 동작하는 웹 흐름을 구현합니다.

목표는 “모든 금융상품을 한 번에 구현”하는 것이 아니라, 회계 근거가 있는 핵심 흐름을 끝까지 동작시키고 검증 가능하게 만드는 것입니다.

04 Screen Design

화면 설계

화면배치 요소설계 의도
대시보드현재 상태, 다음 행동, 단계별 바로가기처음 접속한 사용자가 어디서 시작해야 하는지 바로 알 수 있게 합니다.
공정가치 평가계약 입력 폼, 평가 결과 카드, 계약/평가 이력계약 등록과 평가 결과를 같은 화면에서 확인합니다.
헤지 지정지정 폼, 실시간 적격요건 패널, 지정 이력K-IFRS 6.4.1 조건을 입력 중 바로 확인할 수 있게 합니다.
유효성 테스트관계 선택, 변동액 입력, 결과 게이지, 조치 권고PASS/WARNING/FAIL과 회계적 의미를 한 화면에서 설명합니다.
자동 분개관계별 필터, 분개 테이블, OCI/P&L/BS 태그, 다운로드심사자가 분개가 어떤 회계 근거로 생성됐는지 바로 확인합니다.
버튼 흐름은 공정가치 평가 → 헤지 지정 → 유효성 테스트 → 자동 분개 순서로 맞췄습니다.

06 Technical Considerations

기술적 고려사항

항목선택이유
프런트엔드React + Vite + TypeScript복잡한 입력 폼과 결과 화면을 빠르게 분리하고 타입 오류를 줄이기 위해 선택했습니다.
백엔드Spring Boot + Java금융권에서 익숙한 구조이며 Validation, Transaction, JPA 구성이 안정적입니다.
DBPostgreSQL회계 데이터의 정합성과 트랜잭션 처리를 위해 관계형 DB를 사용했습니다.
배포Vercel + Render StarterAWS보다 설정 부담이 작고, 제출용 링크를 빠르게 안정화할 수 있습니다. Render는 Starter로 올려 무료 플랜의 슬립 지연을 제거했습니다.
AWS 비교이번 PoC에서는 미선택EC2/RDS/VPC 설정은 운영화에는 적합하지만, 취업 포트폴리오 제출 전 빠른 시연 안정화에는 과합니다.
AI/RAGClaude + Codex + K-IFRS 지식 조회회계 근거 확인 → 구현 → 테스트 → 문서화 흐름을 빠르게 반복하기 위해 사용했습니다.
외부 서비스 연동

Vercel은 정적 프런트엔드 배포, Render는 Spring Boot 백엔드와 PostgreSQL 운영에 사용했습니다. 프런트엔드는 VITE_API_BASE_URL, 백엔드는 DB_URL과 APP_CORS_ALLOWED_ORIGINS 환경변수로 환경을 분리했습니다.
