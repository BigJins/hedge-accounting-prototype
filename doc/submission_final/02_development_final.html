00 Technology Rationale

기술 선택 이유

이 프로젝트의 기술 선택 기준은 짧은 기간 안에 회계 도메인 로직을 정확히 구현하고, 외부 심사자가 직접 접속해 확인할 수 있는 웹 프로토타입을 안정적으로 제공하는 데 두었습니다.

기술선택 이유
React + Vite공정가치 평가, 헤지 지정, 유효성 테스트, 자동 분개처럼 단계가 분명한 업무 화면을 컴포넌트 단위로 빠르게 구성하기 위해 사용했습니다. Vite는 반복 수정과 빌드 속도가 빨라 PoC 개발에 적합했습니다.
TypeScript헤지 유형, 위험회피수단 유형, 분개 유형처럼 값의 의미가 중요한 필드를 타입으로 고정해 화면과 API 사이의 실수를 줄이기 위해 사용했습니다.
Spring Boot + Java금융권 백엔드에서 익숙한 구조이며, REST API, Validation, Transaction, JPA 연동을 안정적으로 구성할 수 있어 선택했습니다. 금액 계산은 BigDecimal 기반으로 처리했습니다.
PostgreSQL헤지관계, 평가 이력, 유효성 테스트, 분개 이력은 정합성과 트랜잭션이 중요한 관계형 데이터이므로 PostgreSQL을 사용했습니다.
Vercel + Render프런트엔드는 Vercel, 백엔드와 PostgreSQL은 Render로 분리해 배포했습니다. Render는 Starter 인스턴스로 올려 무료 플랜의 슬립 지연을 제거했습니다.
RAG + AI AgentK-IFRS 조항 확인, 요구사항 정리, 구현, 검증, 문서화를 역할별 에이전트 흐름으로 나누어 진행했습니다. 회계 근거를 먼저 확인하고 코드와 테스트로 닫는 검증 루프를 만들기 위한 선택입니다.
AWS 대신 Render를 선택한 이유

AWS EC2/RDS는 운영화에는 적합하지만 VPC, 보안그룹, 인스턴스 관리, DB 연결 설정까지 필요합니다. 본 프로젝트는 취업용 웹 프로토타입 제출이 목적이므로 배포 속도와 링크 안정성을 우선했습니다.

02 Domain Engine

핵심 도메인 엔진

FX Forward

IRP 기반 이론 선물환율, 할인계수, 공정가치 변동을 계산합니다.

IRS FVH

IRS 평가손익과 고정금리채권 장부조정을 K-IFRS 6.5.8 흐름으로 처리합니다.

Dollar-offset

경제적 관계와 참고 범위를 분리해 PASS/WARNING/FAIL을 판정합니다.

엔진구현 내용근거
FX Forward 평가이론 선물환율, 현가계수, 공정가치 계산K-IFRS 1113 Level 2
CFH Lower of Test유효분 OCI, 비효과분 P/L 분리K-IFRS 1109 6.5.11
IRS FVH 분개수단 평가손익과 피헤지항목 장부조정K-IFRS 1109 6.5.8
IRS FVH 상각장부금액 조정 잔액의 후속 상각K-IFRS 1109 6.5.9
80~125%는 IFRS 9에서 합격선이 아닌 참고 범위로 처리했습니다. FAIL 판정은 경제적 관계 훼손 등 실질 조건에 연결했습니다.
